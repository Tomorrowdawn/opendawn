# design.md — python-purist 演进方向

## 诊断

### 当前 skill 做对了什么

- 27 个 concrete case study（坏代码 → 好代码对比），给 agent 提供了模式匹配的样本
- 7 条机械可检查的原则 + checklist，agent 可以逐项打勾
- 强制性的 workflow 步骤顺序

这些都是**左栏**的事——在 agent 已知的认知范围内提醒它。有效。

### 当前 skill 解决不了什么

Agent 写出荒谬方案的根本原因不是缺规则，而是：

1. **频道不对**：agent 用实现语言（chunk、final、过滤逻辑）思考，人类用应然模型（数据从哪来、流向哪里、为什么会有两份）思考
2. **在错误层面修问题**：协议层的 bug 在渲染层修，数据源头的问题在消费端打补丁
3. **自己发明前提**：agent 脑补出"SSE 数据和 DB 数据可能不一致"这种不存在的风险，然后围绕它造一整套防御体系
4. **缺乏比例感**：修一个 2 行的 bug 引入 50 行的架构

这些问题的根源是 agent 缺的不是规则，是**对"这个世界本该怎样"的理解**——也就是应然模型、第一性原理。这不是 skill 能教的（那是 AGI 级别的事）。

### 结论

**Skill 的边界是 concrete pattern matching + mechanical checks。不要试图让 agent 变聪明。**

正确的做法遵循西比拉系统的逻辑：承认算法的天花板，把"人是怎么想的"直接编码进 skill——不是作为规则，而是作为**更多的、更细粒度的场景参考实现**。Agent 是胶水层，把已有模式落地到具体场景。

## 方案：场景参考实现（Scenario Reference Implementations）

### 核心思路

不是教 agent 推理，而是给它足够多的正确样板。Agent 遇到新问题时，不是在真空中推理出方案——而是**匹配到最近的参考实现，比对差异，做最小调整**。

这和现有 case study 的区别：

| | case study（现有） | scenario（新增） |
|---|---|---|
| 形式 | 坏代码 vs 好代码对比 | 完整可用的参考实现 |
| 粒度 | 单个反模式（如 hasattr 滥用） | 原子交互模式（如 SSE 单工流） |
| 目的 | 识别和避免错误 | 直接匹配和复制 |
| 推理需求 | 需要理解为什么坏 | 几乎不需要——照做就行 |

### 原子场景（不按业务分，按交互结构分）

大多数业务场景是 5-10 个原子场景的组合。覆盖 20-30 个原子场景即可匹配绝大多数实际需求。

候选列表：

| 原子场景 | 核心问题 | 典型踩坑 |
|----------|---------|---------|
| `unidirectional-stream` | 一方向另一方持续发送数据块 | 结束信号设计、乱序、背压 |
| `request-response` | 请求-响应，可能异步 | 超时、重试、幂等 |
| `data-sync` | 两端数据保持一致 | 谁是真源、冲突解决、全量 v 增量 |
| `polling-vs-push` | 数据变化的通知方式 | 轮询浪费、推送丢消息 |
| `local-first` | 本地操作，后台同步 | 乐观更新、回滚、冲突 |
| `auth-guard` | 请求链路中的鉴权点 | 放在网关还是服务、token 刷新 |
| `pagination` | 分页获取大量数据 | 游标 v 偏移、数据变更时的稳定性 |
| `task-queue` | 异步任务提交与结果获取 | 幂等、重试、超时、结果存储 |
| `file-upload` | 文件上传与处理状态 | 分片、断点续传、处理中状态 |
| `rate-limit` | 调用频率控制 | 令牌桶 v 滑动窗口、错误码设计 |
| `circuit-breaker` | 下游故障时的熔断 | 半开状态、恢复策略 |
| `cache-strategy` | 缓存读写策略 | cache-aside v write-through、失效 |
| `idempotency` | 重复请求的安全处理 | 幂等键、过期、并发 |
| `event-driven` | 事件发布与消费 | 顺序保证、重复消费、死信 |
| `saga` | 分布式事务补偿 | 补偿步骤设计、最终一致性 |
| `feature-flag` | 功能开关 | 灰度策略、开关层级、技术债务 |
| `batch-processing` | 批量数据处理 | 分片、断点、进度、部分失败 |
| `webhook` | 外部回调接收 | 签名校验、重试、幂等 |
| `import-export` | 数据导入导出 | 格式映射、校验行级错误、大文件 |

### 每个原子场景文件包含

```
scenario/
└── unidirectional-stream/
    ├── README.md         # 场景描述、核心问题、与其他场景的组合方式
    ├── backend.py        # 后端参考实现（伪代码 + 关键路径，不追求可运行）
    └── frontend.ts       # 前端参考实现
```

README.md 结构：

1. **场景定义**：这是什么交互模式？一方是什么角色，另一方是什么角色？
2. **应然模型**：数据从哪来、流向哪里。哪些事按常理不应该发生？
3. **常见错误写法**（素材来自实际踩坑）：
   - 结束信号设计不当（final chunk 可能乱序）
   - 在消费端修协议层的问题
   - 重复获取同一份数据
4. **参考实现**：伪代码 + 关键决策注释
5. **与其他场景的组合**：比如 unidirectional-stream 常与 data-sync（重连后补数据）、auth-guard（流的鉴权）组合

### 和现有 skill 的关系

场景参考实现不替代现有 case study 和 best-practice。三者的关系：

```
best-practice/   → 原则层：什么是对的，为什么（"组合优于继承"）
case-study/      → 反模式层：具体的错误 vs 正确对比（"inheritance-vs-composition.md"）
scenario/        → 实现层：完整场景的参考代码（"unidirectional-stream/"）
```

Agent 的工作流：

```
1. 识别场景 → 匹配最近的 scenario
2. 参考 scenario 的结构设计自己的代码
3. 对照 best-practice 检查原则
4. 对照 case-study 避免已知反模式
5. 运行 mechanical checks
```

## 自动化思路

### 问题：人工维护场景的成本

20-30 个场景，每个场景的参考代码需要覆盖多语言、多框架，人工维护不可持续。

### 方向：爬虫 + LLM 模式合并

```
GitHub Python 仓库
    ↓ 爬虫（按 star / topic 筛选高质量仓库）
具体场景的实现代码（同一场景的 N 种写法）
    ↓ LLM（聚类 + 去重 + 提取共性）
该场景的"最小正确实现"（伪代码）
    ↓ LLM（识别差异点）
该场景的常见变体 + 选型决策树
    ↓ 人工审核
最终入库 scenario/
```

这不是一次性工程，而是一个**持续运行的 pipeline**：

1. 每周爬取新增的高质量仓库
2. LLM 提取新发现的模式变体
3. 合并到已有场景中（或创建新场景）
4. 人工审核变更

### 可行性

- LLM 做"从多个实现中提取最小公分母"是当前能力的强项
- 不需要 LLM 理解语义——只需要识别结构相似性
- 人工只做审核，大幅降低维护成本
